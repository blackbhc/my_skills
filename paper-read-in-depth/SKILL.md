# paper-read-in-depth

## TL;DR（快速参考）

```
输入: 论文路径/arXiv/DOI
输出: 桌面 Markdown + HTML 双格式（语言随论文）

7 步流程:
  Phase 1: 获取论文内容（PDF/arXiv/DOI → 提取全文）
  Phase 2: 背景研究（已有知识判断 → 子智能体嵌套阅读 → 互联网检索兜底）
  Phase 3: 主线 + 核心证据（摘要+结论+引言末段 → 关键实验/图表/推理）
  Phase 4: 技术细节 + 辅助证据（实验方法/辅助实验/理论）
  Phase 5: 价值评估（进步性/不足/横向对比/同行反响）
  Phase 6: 生成文件 → ~/Desktop/<标题缩写>-summary.md
  Phase 7: 公式审查 → 检查输出 Markdown 中公式的 LaTeX 语法与转写准确性，有误直接修正

关键区分:
  核心证据 = 没有它结论不成立 | 辅助证据 = 增强可信度但非必需
```

---

## Purpose

帮助研究者系统性地深度阅读一篇学术论文，自动完成内容获取、背景调研、主线提取、技术分析、价值评估，生成结构统一的 Markdown 总结和 HTML 版本。总结语言自动匹配论文语言（英文论文→英文总结，中文论文→中文总结）。

本 skill 的设计哲学：**先主线后细节，先证据后判断**。Agent 不会逐字翻译论文，而是像资深研究者一样先抓住骨架、再补充血肉、最后给出批判性评价。

---

## Scope

### What this skill does

- 获取论文内容（PDF / arXiv / DOI）
- 识别并补充背景知识（通过子智能体嵌套阅读或互联网检索）
- 提炼论文核心贡献、关键实验证据、推理链
- 分析技术实现细节和辅助实验
- 提供基于证据的批判性价值评估
- 检查输出 Markdown 中公式的 LaTeX 语法与转写准确性，有误直接修正
- 生成结构统一的 Markdown 总结文件，并自动转化为 HTML（语言随论文）

### What this skill does NOT do

- 不翻译全文（需要时使用 `arxiv-paper-translator` 等翻译专用 skill）
- 不审查论文本身的数学公式正确性（量纲、极限行为、推导逻辑等），仅检查 agent 转写/重现的公式是否准确
- 不冒充正式同行评审意见
- 不评估科学结论的最终真伪
- 不处理扫描版 PDF（需外部 OCR 预处理）
- 不为用户撰写新论文

---

## Use When

### Trigger conditions

- 用户给出了具体的论文来源（PDF 文件路径、`arxiv.org/abs/XXX` 链接、DOI 链接）
- 用户明确要求"读/分析/总结/概述"一篇论文
- 用户说"帮我看看这篇论文"、"这篇 paper 讲了什么"
- 用户需要快速理解论文的贡献、方法和领域定位

### Do NOT use when

- 用户只是询问某篇论文的名字或引用格式（一次 `web_fetch` 即可）
- 用户要做的是写论文而非读论文
- 用户只需要翻译论文（用翻译专用工具）
- 用户给的是非学术内容（新闻、博客、产品文档）

## Workflow

### Phase 1: 获取论文内容

根据输入类型选择获取策略。**目标：提取标题、作者、摘要、引言末段、结论、实验核心数据。**

#### 1a. PDF 文件

```
1. 用 Read 工具读取文件（大文件先 offset/limit 分片）
2. 如 PDF 正文不可读，用 Bash 调用:
   - pdftotext <path> -   (无 pdftotext 则 pip install pdfplumber + Python 脚本)
3. 按优先级提取: 标题 → 摘要 → 引言末尾 → 结论 → 实验章节
```

#### 1b. arXiv 链接

```
1. web_fetch https://arxiv.org/abs/XXXX.XXXXX → 标题、作者、摘要、bib
2. 如有 HTML 全文: web_fetch https://arxiv.org/html/XXXX.XXXXX
3. 否则尝试 PDF: web_fetch https://arxiv.org/pdf/XXXX.XXXXX.pdf
```

#### 1c. DOI 链接

```
1. web_fetch https://doi.org/XXXX → 解析出版页面
2. 提取标题、摘要、元数据
3. 如出版页面有全文: 继续读取
4. 如无全文: 补充搜索 arXiv 镜像或 Semantic Scholar API
```

#### 上下文限制时的优先级策略与长篇拆分

**标准论文（<10 页）— 优先级策略:**
```
优先级 高 → 低:
摘要 → 引言末 2-3 段 → 结论 → 主实验表格 → 方法核心段落 → 其余章节
```

**长论文（≥10 页）— 按章节拆分为独立段解读:**

对于 10 页及以上的长论文，全文一次性读取会超出上下文限制。此时按以下策略将论文**拆分为独立部分逐一解读**：

1. 先获取论文目录/章节标题，确认整篇结构
2. 按粒度拆分为独立段：**Abstract** → **Introduction** → **Method** → **每个 Experiment/Result 子章节** → **Conclusion**
3. 每个独立段**分别执行 Phase 2-5 的分析流程**，生成该段的小结
4. 各段小结最后合成为完整总结文件

**执行原则:**
- 每个独立段解读时**仅关注该段内容**，不跨段引入信息
- Abstract + Introduction 段优先解读，建立论文主线假设
- 后续 Method/Experiment 段用于验证和细化主线
- 段间关联与矛盾在合成阶段统一处理
- 独立段本身仍超标时，对该段按子节再次拆分

**合成规则:**
- 输出为单一文件，不含分段标记
- 「研究背景」← Introduction 段 + Phase 2 背景调研
- 「论文主线」← Abstract + Introduction 段为核心
- 「技术细节」← Method + Experiment 段为核心
- 「价值评估」← 综合所有段的解读结果

#### 错误处理速查

| 情况 | 处理 |
|------|------|
| PDF 文件不存在 | 提示用户确认路径，停止流程 |
| web_fetch 返回 403/404 | 尝试替代源（如 arXiv 镜像、Google Scholar），仍失败时告知用户 |
| PDF 正文为扫描版 | 告知用户需要 OCR，暂停流程等待指示 |
| DOI 无公开全文 | 基于摘要+引言继续分析，在输出中标注"全文受限" |
| 论文超过上下文窗口 | 严格执行优先级策略分段读取，不丢失结论和实验数据 |

### Phase 2: 背景研究

读完论文后，**识别你需要但当前不具备的背景知识**，然后按决策树执行调研。

#### 决策树

```
对于每个背景点:
├─ 我已有充分知识？ → 直接用于分析，跳过调研
├─ 是另一篇论文？ → load_skill("dispatching-parallel-agents") 派生子任务
│     └─ 子任务阅读该论文，返回中文背景摘要
├─ 是概念/理论/术语？ → web_fetch 搜索
│     ├─ Wikipedia / 百度百科 → 概念定义
│     ├─ 综述论文（survey/review）→ 领域全景
│     └─ 技术博客（lilianweng.github.io 等）→ 技术解读
└─ 用户提供了额外资料？ → Read 读取用户指定的文件
```

#### 子智能体（dispatching-parallel-agents）调用规范

```
命令: load_skill("dispatching-parallel-agents")

子任务构造:
  - 指明论文标题、作者
  - 具体问题（如"这篇论文的核心方法是什么？""其实验对本文有什么支撑？"）
  - 输出要求: 300-500 字中文摘要，聚焦与主论文的关系

结果处理:
  - 子任务返回后并入 Phase 2 分析，写入输出文件的「研究背景」章节
```

---

### Phase 3: 论文主线 + 核心证据链

**核心问题 → 核心方法/结论 → 核心证据 → 推理链。**

#### 3.1 提炼主线

聚焦三段黄金位置：
- **摘要（Abstract）**：核心问题 + 主要发现
- **引言末段（Introduction last 2-3 paragraphs）**：贡献声明 + 论文路线图
- **结论（Conclusion）**：总结了什么、意味着什么

回答三个问题：
1. 这篇论文要解决什么问题？（motivation）
2. 他们提出了什么新方法/新结论？（contribution）
3. 高层思路是什么？（high-level idea，一两句话讲清核心机制）

#### 3.2 提取核心证据链

找出**没有它结论就不成立**的证据：

| 证据类型 | 查找位置 | 提取要点 |
|---------|---------|---------|
| **关键实验结果** | 主实验 Table/Figure | 与 SOTA/baseline 的核心数值对比，提升幅度 |
| **Key Figure / Key Plot** | 论文核心图表 | 图表展示了什么现象/趋势/对比，如何支撑主张 |
| **消融实验（核心组件）** | Ablation section | 每个设计选择/组件的量化贡献 |
| **推理过程** | Discussion/Analysis | "因为观察到 X → 所以推理出 Y" 的逻辑链 |

> **筛选标准**：每列出一条证据时，问自己"删除这条证据，结论还站得住吗？"——站不住的就是核心证据。

#### 3.3 写到输出时

```
2.3 核心证据链:
  - 关键实验结果: [具体数字，如 "Table 2 中，本文方法在 X benchmark 上达到 89.3% 准确率，
    超出 SOTA (基线B) 3.1 个百分点"]
  - Key Figure: [描述，如 "Figure 3 展示了方法在训练数据量变化下的收敛曲线:
    横轴训练步数 / 纵轴 loss / ... 关键趋势是..."]
  - 必要推理: [从证据到结论的链条，如 "消融实验显示模块 A 贡献 +2.1%，
    模块 B 贡献 +1.5%，二者叠加非冗余"]
```

### Phase 4: 技术细节 + 辅助证据

#### 4.1 实验/测试方法

- 数据集描述（名称、规模、划分方式）
- 评价指标（accuracy / F1 / BLEU / 自定义 metric 等）
- 实验设置（硬件、framework、超参、随机种子）
- Baseline 方法（名称 + 引用 + 为何选它们作为对比）

#### 4.2 辅助性证据

与 Phase 3 **核心证据**的区别：辅助证据增强可信度，缺少不推翻主要结论。

| 类型 | 示例 |
|------|------|
| 子组件消融 | 次要模块的逐项分析 |
| 超参敏感性 | 学习率、batch size、层数等的影响曲线 |
| 可视化分析 | attention maps、feature embeddings、case studies |
| 泛化性测试 | 其他数据集、其他领域的迁移效果 |
| 失败案例分析 | 什么情况下方法会失效？原因？ |

#### 4.3 实验难点与解决方案

- 论文在实验/实现中遇到什么困难？
- 用什么巧妙方法解决？
- 是否有值得借鉴的实验设计技巧？

#### 4.4 理论推理方法论

- 数学推导的核心步骤（概念层面，不做逐行验证）
- 核心假设和适用条件
- 方法的形式化定义

#### 4.5 关键技术/理论背景

- 所依赖的关键技术栈原理
- 与此论文最相关的前序工作的核心思想
- 必要的领域基础概念

---

### Phase 5: 价值评估

#### 5.1 进步性与贡献

- 相对于已有方法的增量贡献（incremental vs. novelty）
- 论文声称的贡献是否在实验中被充分验证
- 是否有理论/实践上的突破

#### 5.2 不足与局限性

- **论文自身承认的局限**（通常见 Conclusion 或 Limitations 小节）
- **你发现的潜在问题**（从以下维度审视）：
  - 实验设计是否有漏洞？（缺失对比、数据集选择偏差、评价指标不全）
  - 假设是否合理？（适用条件过强、简化过度）
  - 结论泛化性？（只在特定设置下有效？计算成本合理？）
  - 有无 confounding factors？
- 未回答的问题和未来工作方向

#### 5.3 与相关工作的横向对比

- 与 2-3 篇最相关的前序/同期工作对比
- 对比维度：创新性、实验效果、计算效率、适用范围、可解释性

#### 5.4 同行反响

- 如果已知：引用次数、后续跟进工作
- 如果有：OpenReview 讨论、学术社区评价
- 如果有：知名学者/机构的公开评价
- 如果信息不足：注明"暂无公开可用信息"

### Phase 6: 生成文件（Markdown + HTML）

**6a. 确定输出语言**

在输出前，检测论文正文语言：
- 英文论文 → 英文总结
- 中文论文 → 中文总结
- 中英混合 → 以标题和摘要的主要语言为准

所有 Phase 2-5 的分析内容也应以该语言输出。

**6b. 生成 Markdown 文件**

1. 加载 `templates/paper-summary.md` 获取结构
2. 将 Phase 2-5 的输出填入对应章节
3. 确认: 无空章节、无占位符、无 "${...}" 未替换内容
4. 用 `write` 工具写入文件

**输出路径:**
```
/Users/chenbinhui/Desktop/<论文标题缩写>-<YYYY-MM-DD>.md
```
- 标题缩写取论文标题前 4-6 个英文单词（小写、连字符连接）
- 如标题是中文，取前 4-6 个关键词

**6c. 自动转化 HTML**

Markdown 文件写入后，立即用 `bash` 调用 Python `markdown` 库将其转为 HTML 文件（同名，仅后缀不同）。HTML 需包含基本样式（字体、排版、表格样式等），确保在浏览器中可读。生成后使用 `open` 命令自动在浏览器中打开 HTML 文件。

最后在对话中告知用户 `.md` 和 `.html` 两个文件的路径。

---

### Phase 7: Markdown 公式审查

生成 Markdown 文件后，对输出文件中** agent 写入的所有数学公式**进行逐公式检查，确保 Markdown/LaTeX 公式与论文原文一致。如有差异，**直接修正**输出文件，不在输出中添加审查章节。

> 此阶段仅关注「agent 写对了没有」，不评判论文本身公式的正确性。

#### 检查清单

对输出文件中的每个 **LaTeX 公式（`$...$`、`$$...$$`、`\\(...\\)`、`\\[...\\]`）**，逐一检查：

| 检查项 | 说明 | 修正方式 |
|--------|------|---------|
| **语法正确性** | LaTeX 命令拼写无误？花括号成对？反斜杠未遗漏？ | 直接修正 LaTeX 源码 |
| **公式完整性** | 从论文复制/转写时有无遗漏项、索引、上下标？ | 对照论文补全 |
| **符号一致性** | Markdown 中的符号与论文原文使用的符号一致？（大小写、希腊字母、花体等） | 统一替换为论文中的符号 |
| **转写准确性** | 分数、求和、积分等结构是否准确地从论文排版转为 LaTeX 源码？ | 重写为正确的 LaTeX 结构 |
| **行内/展示模式** | 短公式是否误用 `$$`（导致换行）？长公式是否误用 `$`（渲染异常）？ | 调整分隔符 |

#### 执行方法

1. 扫描输出 Markdown 文件中所有 `$...$`、`$$...$$`、`\\(...\\)`、`\\[...\\]` 包裹的 LaTeX 代码
2. 逐公式对照论文原文，做上述 5 项检查
3. **发现问题直接 `edit` 修改输出文件**（不添加审查备注到文件末尾）
4. 全部通过后不做任何记录

#### 注意事项

- 如果论文本身有额外 material（附录公式、补充材料），Markdown 总结中不需要包含所有公式，仅需确保已写入的公式准确
- 不需要在输出文件中添加"公式审查"章节——修正完即为完成
- 不允许跳过此阶段：即使修改量为零，也必须逐公式扫描一遍

## Error Handling Guide

| 阶段 | 异常 | 处理 |
|------|------|------|
| Phase 1 | PDF 路径不存在 | 告知用户，请求确认，**停止流程** |
| Phase 1 | PDF 为扫描版（无文字层） | 告知用户需要 OCR 预处理，等待指示 |
| Phase 1 | arXiv 服务器无响应 | 重试 1 次 → 仍失败则使用 Semantic Scholar API 获取摘要 |
| Phase 1 | DOI 链接仅返回 paywall | 使用 arXiv / Semantic Scholar 补充，标注"摘要分析" |
| Phase 2 | 子任务引用的论文无法获取 | 改用 web_fetch 搜索该论文的摘要/博客解读 |
| Phase 2 | 互联网不可用 | 跳过 nest 阅读和 web 检索，在输出中标注"基于现有知识分析" |
| Phase 3-5 | 论文缺少某些章节（如无消融实验） | 在对应输出章节标注"该论文未报告此项" |
| Phase 6 | 桌面目录不可写 | 退回到 `/tmp/` 并告知用户 |

## Pitfalls & Common Mistakes

这是 Agent 执行本 skill 时容易犯的错误，请逐条自检：

| # | 典型错误 | 正确做法 |
|---|---------|---------|
| 1 | **逐句翻译论文**，生成冗长无重点的流水账 | 按 Phase 3 先抓骨架（核心问题→方法→结论→关键证据），再做 Phase 4 细节 |
| 2 | **核心证据和辅助证据混为一谈** | 核心证据 = 少了结论不成立；辅助 = 锦上添花 |
| 3 | **Key Figure 只贴标题不解读** | 必须描述：这张图展示什么？横纵轴是什么？趋势说明了什么？ |
| 4 | **价值评估只夸不批** | 必须找出至少 1 个不足或潜在问题 |
| 5 | **Phase 2 跳过已有知识，不做双层检查** | 即使你觉得"知道这个概念"，也要确认你的理解与论文中的用法一致 |
| 6 | **输出路径忘记加日期** | 始终用 `<标题缩写>-<YYYY-MM-DD>.md` |
| 7 | **空章节用占位符填充就提交** | 遇到缺失内容必须标注"该论文未报告此项"，不能留"TODO" |
| 8 | **dispatching-parallel-agents 子任务无明确问题** | 子任务必须带具体问题（如"X 方法的核心公式是什么？"），不能只说"读这篇论文" |
| 9 | **Phase 7 Markdown 公式审查被跳过或走形式** | 必须逐公式扫描输出文件中的所有 LaTeX 源码，检查 5 个维度，不允许说"所有公式正确"后跳过 |

## Outputs

| 输出 | 格式 | 位置 |
|------|------|------|
| 论文总结文件 | Markdown | `~/Desktop/<标题缩写>-<日期>.md` |
| HTML 版本 | HTML | `~/Desktop/<标题缩写>-<日期>.html`（自动打开） |
| 对话摘要 | 自然语言 | 生成文件后在对话中输出各章节要点 |

## Limitations

- **不保证科学结论的正确性** — 提取和分析内容，但不做事实性验证
- **不处理非学术文本** — 新闻、博客、技术产品文档不在此 skill 范围内
- **不生成正式评审意见** — 价值评估仅供参考，不冒充同行评审
- **不对数学推导做符号级验证** — 仅做概念性总结
- **嵌套阅读的子任务质量受限于可用上下文** — 引用论文过长时可能只读到摘要
- **图表解读基于论文中的文字描述** — 无法直接"看懂"图表截图

## Checklist

Agent 完成本 skill 后，逐项自检：

**输入处理:**
- [ ] Phase 1: 正确识别输入类型并成功提取论文内容
- [ ] 如全文受限，已在输出中明确标注

**背景研究:**
- [ ] Phase 2: 对每个需要的背景点执行了至少一种调研策略
- [ ] 使用 dispatching-parallel-agents 时子任务有明确的问题和输出要求

**分析质量:**
- [ ] Phase 3: 主线清晰，核心证据链通过"删掉它结论还成立吗"测试
- [ ] Phase 3: 至少解读了 1 个 Key Figure/Key Plot（有描述而非仅标题）
- [ ] Phase 4: 技术细节覆盖面完整（实验方法、辅助证据、难点、理论）
- [ ] Phase 5: 有至少 1 条批判性评价（不只是夸赞）

**输出质量:**
- [ ] Phase 6a: 已根据论文正文语言确定输出语言
- [ ] Phase 6b: Markdown 文件按模板格式生成，路径正确
- [ ] Phase 6c: HTML 文件已自动生成并在浏览器中打开
- [ ] 无空章节（缺失内容已标注"该论文未报告此项"）
- [ ] 无占位符（`{{}}`、`TODO` 等）
- [ ] 全文语言流畅无歧义

**公式审查:**
- [ ] Phase 7: 已扫描输出文件中所有 `$...$` / `$$...$$` / `\\(...\\)` / `\\[...\\]` 公式
- [ ] Phase 7: 每个公式已完成 5 项检查（语法/完整性/符号/转写/模式）
- [ ] 发现问题已直接 `edit` 修正，未在输出中添加审查章节
- [ ] 未出现"所有公式正确"后跳过逐公式扫描的情况
