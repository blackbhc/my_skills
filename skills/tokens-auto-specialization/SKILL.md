# Skill: tokens-auto-specialization

## Purpose

这是一个**持续激活**的 prompt 构造方法论，不是等用户说「帮我写 prompt」才调用的一次性工具。Agent 在运行期间，每当需要为某个任务构造 prompt（包括自用、委派、工具调用、思维链规划），都应按本 Skill 的框架自动补齐并构造。

## Use When

本 Skill **在整个 Agent 运行期间持续生效**，不依赖用户显式说出「prompt」「提示词」。以下情形自动触发：

- Agent 接到任何任务，需要将模糊指令转化为可执行的 prompt
- Agent 内部委派子任务，需要为目标 Agent/工具构造 prompt
- Agent 感觉当前 prompt 不足以引导模型产出高质量结果
- Agent 规划多步推理，需要为每步构造独立 prompt
- 用户给出场景+用途（无论是否提及「prompt」一词）

**一句话：每当 Agent 需要「写 prompt」——无论给自己、给工具、还是给其他模型——本 Skill 就激活。**

## Inputs

本 Skill 从 Agent 当前上下文中**自动提取**输入，无需用户显式提供：

| 输入 | 来源 | 说明 |
|------|------|------|
| 使用场景 | 用户任务上下文 / Agent 当前目标 | 在什么情境下使用 |
| 用途/目标 | 用户指令 / Agent 子目标 | prompt 要达成什么效果 |
| 目标模型 | Agent 环境 / 配置 | 当前使用的模型，未指定则用通用策略 |
| 约束条件 | 系统上下文 / 安全策略 | 已有的边界条件、格式要求 |

## Workflow

Agent 在构造 prompt 时，按以下步骤执行：

1. **提取场景与目标** —— 从当前上下文明确「谁用」「做什么」「为什么」
2. **补齐隐形需求** —— 推导没说出口的角色定位、上下文环境、痛点、成功标准
3. **推导规则约束** —— 边界条件、质量标准、输出长度/风格限制
4. **定义输出格式** —— 结构、字段、标记语言
5. **模型适配** —— 按下方「模型适配原则」调整
6. **输出双版本** —— 精简通用版（优先）+ 高阶顶配版（需要时）

## Outputs

Agent 在以下时机产出 prompt：
- 接收用户任务后，**内部构造**自用 prompt（精简版，不展示给用户）
- 需要向用户展示 prompt 时，输出**双版本**（精简版在前，`=====` 分隔高阶版在后）
- 向工具/子 Agent 委派任务时，输出精简版 prompt

### 精简通用版

| 维度 | 要求 |
|------|------|
| 定位 | 日常高效使用，跨模型稳定运行 |
| Token 预算 | ≤300 tokens |
| 角色 | 一句话定位 |
| 规则 | ≤5 条核心约束 |
| 输出格式 | 简要指定 |
| 适配 | 通用写法，不依赖特定模型语法 |

### 高阶顶配版

| 维度 | 要求 |
|------|------|
| 定位 | 复杂/专业场景，追求最佳输出质量 |
| Token 预算 | 不限 |
| 角色 | 完整人设：专家定位 + 能力边界 + 表达风格 |
| 思维链 | 显式 CoT 引导步骤 |
| 约束 | 全维度：内容、格式、风格、边界、质量门槛 |
| 输出 Schema | 结构化字段、类型、格式要求 |
| Few-shot | 1–2 个输入输出示例 |
| 容错 | 兜底策略：理解偏差处理、边界情况应对 |

### 输出格式（对用户展示时）

```
【精简通用版】
...(≤300 tokens)

=====

【高阶顶配版】
...(完整结构化 prompt)
```

**硬约束：** 零寒暄、零解释、零铺垫；两个版本都**可直接使用**。内部自用时只生精简版，不展示双版本标记。

## Model Adaptation Principles

不要硬编码特定模型名（模型列表会演进），按以下原则适配：

| 原则 | 做法 |
|------|------|
| **中文优先模型** | 中文 prompt；简化思维链避免 token 浪费；用标题/列表结构替代复杂 JSON |
| **强推理模型** | 可保留完整 CoT；善用长文本和结构化输出能力 |
| **支持 System/User 角色分离** | 将角色定义和规则放入 system prompt，具体任务放入 user prompt |
| **长上下文模型** | 可用 XML 标签辅助结构化（`<task>` `<rules>` `<output>`）；利用多轮对齐能力 |
| **多模态模型** | 指令靠前放置；避免依赖过长纯文本示例 |
| **未知/通用** | 纯文本 Markdown 结构，不依赖任何模型特定语法 |

## Limitations

- 不负责验证用户场景的合理性（如恶意用途应拒绝）
- 不负责执行 prompt 后的结果验证
- 用户输入极端简单（≤5 个字）时可能无法准确推导，需追问确认
- 内部自用时，优先精简版——不要每件事都生成高阶版

## Examples

### 示例 1：内部自用（不展示双版本）

> **用户**：帮我分析这个项目的目录结构是否合理

> **Agent 内部构造（精简通用版）**：
> 你是软件架构分析专家。分析目录结构时：
> 1. 按关注点分离、模块边界、命名规范三个维度评估
> 2. 每个维度给出具体问题点和改进建议
> 3. 用 Markdown 列表输出，先结论后分析
> 4. 不评价代码质量，只看结构

### 示例 2：对用户展示（双版本）

> **用户**：需要一个代码审查 prompt，场景是审查 React TypeScript PR，关注安全、性能、可维护性

> **Agent 输出**：
>
> 【精简通用版】
> 你是资深前端审查员。审查此 PR 时：
> 1. 按安全→性能→可维护性→类型安全顺序逐文件检查
> 2. 每个问题标注优先级（P0/P1/P2）、文件+行号、风险说明、修复建议
> 3. 用 Markdown 表格输出
> 4. 不评价代码风格偏好（如单引号 vs 双引号）
>
> =====
>
> 【高阶顶配版】
> ...(含 CoT、输出 Schema、Few-shot、容错策略的完整 prompt)

## Checklist

输出前自检：

- [ ] 是否需要双版本展示给用户？不需要则只生成内部精简版
- [ ] 精简版是否 ≤300 tokens？
- [ ] 是否零寒暄、零解释、零铺垫？
- [ ] 是否覆盖了隐形需求、规则约束、输出格式三个维度？
- [ ] 模型适配策略是否与目标模型匹配？
- [ ] 高阶版是否包含 Few-shot 和容错策略？
- [ ] 用户明确说「prompt」「提示词」时，展示双版本；否则内部自用
