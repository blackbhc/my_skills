# Skill: tokens-auto-specialization

## Purpose

用户只需给出使用场景 + 用途，自动补齐隐形需求、角色设定、规则约束、输出格式、模型适配，输出**可直接使用**的精简通用版和高阶顶配版两个 prompt。

## Use When

- 用户描述场景并需要 prompt / 提示词
- 用户说「帮我写个 prompt，用于…」
- 用户要求优化或重构现有提示词
- 触发词：提示词、prompt、提示工程、prompt engineering、写prompt、优化提示词

## Inputs

| 输入 | 必需 | 说明 |
|------|------|------|
| 使用场景 | ✅ | 在什么情境下使用（如：客服对话、代码审查、论文润色） |
| 用途/目标 | ✅ | 希望 prompt 达成什么效果 |
| 目标模型 | ❌ | 未指定则输出通用版本，高阶版中标注适配建议 |

## Workflow

收到用户输入后，按以下步骤执行：

1. **补齐隐形需求** —— 推导用户没说出口的角色定位、对话目标、上下文环境、痛点
2. **推导规则约束** —— 边界条件（不能做什么）、质量标准、输出长度/风格限制
3. **定义输出格式** —— 结构（表格/列表/Markdown/JSON）、字段定义
4. **模型适配分析** —— 根据目标模型特点调整（见下方「模型适配原则」）
5. **生成双版本** —— 先输出精简通用版，用 `=====` 分隔后输出高阶顶配版

## Outputs

### 精简通用版

| 维度 | 要求 |
|------|------|
| 定位 | 日常高效使用，跨模型稳定运行 |
| Token 预算 | ≤300 tokens |
| 角色 | 一句话定位 |
| 规则 | ≤5 条核心约束 |
| 输出格式 | 简要指定 |
| 适配 | 通用写法，不依赖特定模型语法 |

### 高阶顶配版

| 维度 | 要求 |
|------|------|
| 定位 | 复杂/专业场景，追求最佳输出质量 |
| Token 预算 | 不限 |
| 角色 | 完整人设：专家定位 + 能力边界 + 表达风格 |
| 思维链 | 显式 CoT 引导步骤 |
| 约束 | 全维度：内容、格式、风格、边界、质量门槛 |
| 输出 Schema | 结构化字段、类型、格式要求 |
| Few-shot | 1–2 个输入输出示例 |
| 容错 | 兜底策略：理解偏差处理、边界情况应对 |

### 输出格式

```
【精简通用版】
...(≤300 tokens)

=====

【高阶顶配版】
...(完整结构化 prompt)
```

**硬约束：** 零寒暄、零解释、零铺垫，直接输出 prompt；两个版本都**可直接使用**。

## Model Adaptation Principles

不要硬编码特定模型名（模型列表会演进），按以下原则适配：

| 原则 | 做法 |
|------|------|
| **中文优先模型**（如国产模型） | 中文 prompt；简化思维链避免 token 浪费；用标题/列表结构替代复杂 JSON |
| **强推理模型** | 可保留完整 CoT；善用长文本和结构化输出能力 |
| **支持 System/User 角色分离** | 将角色定义和规则放入 system prompt，具体任务放入 user prompt |
| **长上下文模型** | 可用 XML 标签辅助结构化（`<task>` `<rules>` `<output>`）；利用多轮对齐能力 |
| **多模态模型** | 指令靠前放置；避免依赖过长纯文本示例 |
| **未知/通用** | 纯文本 Markdown 结构，不依赖任何模型特定语法 |

## Limitations

- 不负责验证用户场景的合理性（如「帮我写个骗人的 prompt」应拒绝）
- 不负责执行 prompt 后的结果验证
- 用户输入极端简单（≤5 个字）时可能无法准确推导，需追问确认

## Examples

### 示例 1：简单场景

> **用户**：帮我写个 prompt，用于每天总结我的待办事项，按优先级排列

> **输出**：
>
> 【精简通用版】
> 你是个人效率助手。根据用户提供的待办事项列表：
> 1. 按优先级（紧急重要 > 紧急不重要 > 重要不紧急 > 不紧急不重要）排序
> 2. 每项标注优先级标签和预计耗时
> 3. 用 Markdown 表格输出，列：优先级 | 事项 | 预计耗时 | 建议执行时段
> 4. 不添加任何解释性文字

### 示例 2：专业场景

> **用户**：需要一个代码审查 prompt，场景是审查 React TypeScript PR，关注安全、性能、可维护性

> **输出**：（精简版 + 高阶版双版本，高阶版含 CoT 引导、输出 Schema、Few-shot、容错策略）

## Checklist

输出前自检：

- [ ] 两个版本是否都以「【精简通用版】」「【高阶顶配版】」标签开头？
- [ ] 精简版是否 ≤300 tokens？
- [ ] 是否零寒暄、零解释、零铺垫？
- [ ] 两个版本是否都**可直接使用**，无需用户二次编辑？
- [ ] 是否覆盖了隐形需求、规则约束、输出格式三个维度？
- [ ] 模型适配策略是否与目标模型匹配？（未指定则用通用策略）
- [ ] 高阶版是否包含 Few-shot 和容错策略？
